How ClaudeCraft Was Actually Built

An industrial worker, no engineering background, building a real selling website with Claude Code — including the brain fog, the sleep tradeoffs, and the parts that went sideways. The honest version, not the highlight reel.

Why I'm Writing This

I work an industrial job. Building this happened in the hours around that — early mornings before a shift, late nights after one, weekends when I had them. I'm telling you that up front because most "how I built my SaaS" content gets written by people who had nothing but time, and that's not realistic for most people reading this.

Some of it was genuinely hard in ways that had nothing to do with code. There were nights I sat down to work on this with real brain fog — tired in a way where reading a single paragraph twice still didn't land. The honest fix wasn't pushing through harder. It was smaller sessions, more often, and being okay with a night where the only thing that got done was one small fix instead of a big feature. Progress that's slow but doesn't stop beats a burst of effort that burns you out for a week afterward. I learned that the hard way, more than once.

I'm not sharing this because I think my story is special. I'm sharing it because I think the actual barrier for most people isn't intelligence or even time — it's believing the technical part is harder than it actually is now, and giving up before finding out it isn't. I genuinely want this to close that gap for you. Not a sales pitch — an honest map of the terrain, including the potholes.

Part 1: Understanding Your Computer's File System

This is the part almost every tutorial skips, and it's exactly the part that quietly stops beginners cold later on.

What a file system actually is

Every file lives inside a folder, and every folder lives inside another folder, all the way back to one root location. A "path" is just the full address of where a file lives — e.g. C:\Users\YourName\Desktop\my-project\index.html tells you exactly where to find it.

Navigating in VS Code

File → Open Folder, select your project. The sidebar now shows everything inside it — click to open, right-click for New File/Folder/Rename/Delete.

Navigating in the terminal

  • cd my-project — moves into a folder named "my-project"
  • cd .. — moves UP one level
  • dir (Windows) or ls (Mac/Linux) — lists what's in your current folder

That's the whole list you actually need to memorize. Claude Code handles the complex commands for you when you ask.

Part 4: What GitHub Actually Is, and Why It Matters

GitHub stores a copy of your project online and keeps a complete history of every change ever made — like a save-game system for your whole project, not just one file. "Git" is the tracking tool itself (built into VS Code already); "GitHub" is the most popular website for hosting those tracked projects.

Why a non-engineer should care: if something breaks, you can go back to any earlier saved point — your work is never permanently lost to one bad change. It's also the standard way to hand your code to a hosting service so it can actually go live.

Sign up free at github.com, click New Repository, then ask Claude Code directly: "help me connect this project to the GitHub repo I just created and push my code to it." You don't need to memorize Git commands.

Part 5: "Vibe Coding" — The Actual Workflow

The real pattern, every single time: describe the feature in plain English. Claude Code reads your existing files first — it doesn't guess. It makes the change and explains what it touched. You check it for real, by actually loading the result and looking. If something's wrong, say exactly what's wrong, the same way you'd correct a person.

Big features get built with the exact same loop as small ones — just more rounds of it. The single most common beginner mistake: asking for five things in one giant request. Ask for one, test it, then ask for the next.

"Vibe" doesn't mean skip reviewing. The loose, conversational feel only works long-term if you still review every change, especially anything touching money, passwords, or user data.

Part 6: Connecting Stripe for Real Payments

Stripe is a payment processor — it handles someone's credit card safely so you never touch raw card numbers yourself. "Checkout" is the page Stripe shows your customer to pay. A "webhook" is Stripe notifying your server after a payment succeeds, so your code knows to deliver the product.

Sign up at stripe.com, toggle to Test mode first, copy your test Secret key (never paste a real key into any AI chat — keep it in environment variables instead). Test with card number 4242 4242 4242 4242 before ever switching to your live key.

Part 7: Deploying for Real — Railway & GitHub

ClaudeCraft runs on Railway (railway.app) — it deploys straight from a GitHub repo with minimal configuration. Create a free account, then ask Claude Code: "help me deploy this project to Railway." Add secret keys in Railway's own Variables tab — never inside your code files.

Part 8: Setting Up Your Own Scheduled Agent

ClaudeCraft has a small companion script that runs automatically once a day, pulls fresh information, and publishes a new article with zero manual work. A "cron job" is just an instruction to a hosting service: "run this script automatically, on this schedule, forever." You don't need to memorize the schedule syntax — ask Claude Code to write it for you.

Keep it simple on purpose: one script with a few clearly-named functions inside it is easier to maintain than splitting every task into its own separate service.

Getting Through the Tough Spots

Brain fog is real — plan around it instead of fighting it. On low-energy nights, shrink the task instead of pushing through. "Fix this one small thing" is a completable session. "Build the whole feature" on no sleep usually isn't.

Protecting sleep and your actual job. This got built around a real industrial job, not instead of one. Smaller, more frequent sessions beat rare giant ones that eat sleep before a shift. Consistency is the actual goal, not speed.
